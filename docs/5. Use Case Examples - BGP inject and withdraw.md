# Use Case Examples - BGP inject and witdraw

Using the gathered knowledge and defined building blocks to create some examples.

The developed examples are:
- Network examples:
  - BGP inject and witdraw.
- Operational example:
  - Ticket to SOC.

Usage:
- Create the Violation Action script.
- Define a Violation Action and include the created script.
- Include the Violation Action to the protocol at Zone > Services. Add it to the desired zone level where the script should execute.
- Be sure to keep an eye on `tail -f /var/log/tps.log` at your syslog server.
- Use a DDoS stress test to get to the required load to hit the level thresholds.

## Network Examples: BGP inject and withdraw using measured `DDOS_EVENT` behavior

This section describes the recommended configuration approach that worked in our lab.  
The key success factor is that we first **measured** where each `DDOS_EVENT` appears, and then attached the scripts to the correct attachment points.

### Goal

- Inject a BGP prefix when the zone **escalates**
- Withdraw the BGP prefix when the zone **de-escalates**
- Avoid trial and error by using probe results to determine where to attach the scripts

### Prerequisites

- External syslog is configured and you can see TPS logs on Ubuntu in `/var/log/tps.log`
- The Python event-capture probes have been executed and you have confirmed where you observe:
  - `DDOS_ZONE_ESCALATION`
  - `DDOS_ZONE_DE_ESCALATION`
- You have working `bgp_inject.py` and `bgp_withdraw.py` scripts (see previous section)

### Step 1 Confirm event mapping using probes

Attach event-capture probes at the **zone level** (L1, L2, L3).  
Run one full test flow:

1. Scale up until the zone reaches level 3  
2. Stop traffic and wait until the zone returns to level 0  

Confirm the following mapping from syslog output:

- Which level hook(s) produce `DDOS_ZONE_ESCALATION`
- Which level hook(s) produce `DDOS_ZONE_DE_ESCALATION`

In our ICMP zone test we observed:

- `DDOS_ZONE_ESCALATION` commonly delivered via L1 and L2 callbacks during scale up
- `DDOS_ZONE_DE_ESCALATION` delivered via the L3 callback when stepping down from level 3 to level 2
- `DDOS_ZONE_DE_ESCALATION` also delivered via L2 and L1 callbacks during further step-down (2→1, 1→0)

Your environment may vary, so always confirm with probes first.

### Step 2 Deploy BGP inject and withdraw violation action scripts.
At A10 Control install:
- Below execute scripts.
- Attach them to separate `Violation Actions`.
  
**Note:** attaching them to the zone service config is described in the next steps.

**BGP_inject**
The full `bgp_inject` script can be found here: [bgp_inject.py](../scripts/bgp_inject.py)

Give the script a name, eg. `bgp_inject.py` and deploy it on A10 Control.
Ensure you change filled in defaults like: Syslog server IP, AXAPI host, AXAPI user and password, BGP ASN and PREFIX.


**BGP_witdraw**
The `bgp_witdraw` script can be found here: [bgp_witdraw.py](../scripts/bgp_witdraw.py) 
Give thescript a name, eg. `bgp_witdraw.py` and deploy it on A10 Control.
Ensure you change filled in defaults like: Syslog server IP, AXAPI host, AXAPI user and password, BGP ASN and PREFIX.


### Step 3 Configure BGP inject at the correct zone level hook

Attach `bgp_inject.py` at the zone level hook where you **consistently** observe:

- `DDOS_EVENT == DDOS_ZONE_ESCALATION`

In our lab this was typically the L2 zone level hook (and sometimes L1 depending on the transition).

Expected outcome:

- When the zone escalates, the script runs and injects the configured prefix
- The syslog output shows `stage=start`, `stage=action`, `stage=end` with `ok=true`
- The prefix becomes visible in the BGP table and BGP configuration

### Step 4 Configure BGP withdraw at the correct zone level hook

Attach `bgp_withdraw.py` at the zone level hook where you observe:

- `DDOS_EVENT == DDOS_ZONE_DE_ESCALATION`

In our lab:

- The L3 zone level hook was a reliable place to withdraw the prefix when stepping down from level 3 to level 2

If you also want cleanup during further step-down transitions (2→1 and 1→0), then either:

- attach `bgp_withdraw.py` additionally at L2 and or L1 zone level hooks, or
- use the combined inject+withdraw script at multiple zone levels and let it decide based on `DDOS_EVENT`

Expected outcome:

- When the zone de-escalates, the script runs and withdraws the configured prefix
- The syslog output shows `stage=start`, `stage=action`, `stage=end` with `ok=true`
- The prefix is removed from both the BGP configuration and the BGP table

### Step 5 Validate behavior using observability

Validation should always be done in two ways:

1. Syslog execution trace  
   Confirm the scripts are executed and whether they acted:
   - `stage=start` includes `DDOS_EVENT` and `will_act=true|false`
   - `stage=action` confirms inject or withdraw
   - `stage=end` confirms result and verification flags

2. Device state verification  
   Confirm the prefix is present or absent via:
   - `show ip bgp`
   - `show running-config | section router bgp`

Optionally, use `ddos_watch_human.log` to see a time-aligned snapshot of:
- zone level changes
- script execution
- BGP prefix presence

### Why this configuration works

- Threshold-based attachment points tend to produce `DDOS_INDICATOR_EXCEED` and are best for indicator threshold actions
- Zone level attachment points produce lifecycle events such as `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`
- By measuring where lifecycle events appear in your environment, you can attach scripts to the correct hook and get predictable automation

### Summary

1. Use probes to discover where lifecycle events are delivered  
2. Attach `bgp_inject.py` to the zone level hook that delivers `DDOS_ZONE_ESCALATION`  
3. Attach `bgp_withdraw.py` to the zone level hook that delivers `DDOS_ZONE_DE_ESCALATION`  
4. Validate with syslog output and BGP state checks
