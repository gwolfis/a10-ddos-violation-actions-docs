# Step-by-step: Building Observability for Violation Action Scripts (Syslog + ddos_watch)

This page explains how to build a practical observability pipeline for **Violation Action → Execute Script** on A10 TPS (Mitigator). The purpose is to get reliable feedback beyond `Script: … Running/Success` by capturing device state via **AXAPI** at the moment events occur.

The design is simple:

1. Forward Mitigator logs to an external Ubuntu syslog server.
2. Run a watcher (`ddos_watch`) on Ubuntu that tails the syslog file.
3. When a trigger is observed (level change or script event), the watcher calls AXAPI (`clideploy`) and writes snapshots:
   - A structured JSONL log
   - A human-readable text log (each entry begins with a timestamp)

## Prerequisites

- Ubuntu server reachable from the TPS management interface
- TCP/UDP connectivity allowed:
  - TPS → Ubuntu: UDP 514 (syslog)
  - Ubuntu → TPS: HTTPS 443 (AXAPI)
- Python 3 installed on Ubuntu (no pip required)
- Root access on Ubuntu (for rsyslog + systemd setup)

Throughout this guide, adjust the IPs:

- TPS (Mitigator) management IP: `10.25.12.132`
- Ubuntu syslog + watcher host: `10.25.12.140`

## Step 1: Configure Ubuntu as a syslog server (rsyslog)

### 1.1 Install rsyslog

```bash
sudo apt update
sudo apt install -y rsyslog
```
### 1.2 Enable syslog reception on UDP 514 (and optionally TCP)

Edit the rsyslog main config.

```bash
sudo vim /etc/rsyslog.conf
```
Ensure these lines exist and are not commented.

```conf
module(load="imudp")
input(type="imudp" port="514")

module(load="imtcp")
input(type="imtcp" port="514")
```
### 1.3 Create a dedicated logfile for TPS logs

Create a new config file.

```bash
sudo vim /etc/rsyslog.d/30-tps.conf
```

Use this content (filter by TPS source IP).
```conf
$template RemoteFmt,"%timegenerated% %fromhost-ip% %hostname% %syslogtag%%msg%\n"

if $fromhost-ip == '<<mitigator-mgmt-ip>>' then /var/log/tps.log;RemoteFmt
& stop
```
Create the logfile with correct permissions.

```bash
sudo touch /var/log/tps.log
sudo chown syslog:adm /var/log/tps.log
sudo chmod 640 /var/log/tps.log
```
Restart rsyslog.
```bash
sudo systemctl restart rsyslog
sudo systemctl enable rsyslog
sudo systemctl status rsyslog --no-pager
```
### 1.4 Open firewall (if using UFW)
```bash
sudo ufw allow 514/udp
sudo ufw allow 514/tcp
sudo ufw status
```
### 1.5 Validate reception

Watch the log.
```bash
sudo tail -f /var/log/tps.log
```
Validate packets are arriving.
```bash
sudo tcpdump -ni any port 514
```
If tcpdump shows packets but /var/log/tps.log stays empty.
```bash
sudo rsyslogd -N1
sudo journalctl -u rsyslog -n 100 --no-pager
```

## Step 2: Configure TPS to forward syslog to Ubuntu

On the Mitigator (TPS), configure remote syslog using the management port.
```Plain Text
configure terminal
logging syslog information
logging host <<syslog-server-mgmt-ip>> use-mgmt-port
end
write memory
```

Verify on TPS.
```Plain text
show running-config log
```
You should see.
```Plain text
logging syslog information
logging host <<syslog-server-mgmt-ip>> use-mgmt-port
```

Now you should see Mitigator logs appearing in `/var/log/tps.log` on Ubuntu.


## Step 3: Install ddos_watch on Ubuntu

`ddos_watch` tails `/var/log/tps.log`. When it sees relevant events, it calls AXAPI `clideploy` and logs results.

### 3.1 Create the install directory
```bash
sudo mkdir -p /opt/ddos-watch
sudo chown -R ubuntu:ubuntu /opt/ddos-watch
```
### 3.2 Create the watcher script

Create the file.
```bash
sudo vim /opt/ddos-watch/ddos_watch.py
```
The full script can be found here: [ddos_watch.py](scripts/ddos_watch.py)

Paste the full script below. Adjust `TPS_HOST`, `TPS_USER`, `TPS_PASS`, and optionally `EXPECTED_PREFIX`.      

Make it executable.
```bash
sudo chmod 755 /opt/ddos-watch/ddos_watch.py
```
### 3.3 Create output files
```bash
sudo touch /var/log/ddos_script_observability.jsonl /var/log/ddos_watch_human.log
sudo chown ubuntu:ubuntu /var/log/ddos_script_observability.jsonl /var/log/ddos_watch_human.log
sudo chmod 644 /var/log/ddos_script_observability.jsonl /var/log/ddos_watch_human.log
```
### 3.4 Ensure the service user can read /var/log/tps.log

On Ubuntu, /var/log/tps.log is owned by syslog:adm. Add user ubuntu to group adm.
```bash
sudo usermod -aG adm ubuntu
```
Log out and log in again (or reboot) so group membership applies.

## Step 4: Run ddos_watch as a systemd service
### 4.1 Create the unit file
```bash
sudo vim /etc/systemd/system/ddos-watch.service
```
The full service file can be found here: [ddos-watch.service](scripts/ddos-watch.service)

### 4.2 Enable and start
```bash
sudo systemctl daemon-reload
sudo systemctl enable --now ddos-watch
sudo systemctl status ddos-watch --no-pager
```
### 4.3 Watch logs

Service logs.
```bash
sudo journalctl -u ddos-watch -f
```

Human readable timeline.
```bash
sudo tail -f /var/log/ddos_watch_human.log
```
Structured JSONL.
```bash
sudo tail -f /var/log/ddos_script_observability.jsonl
```

## Step 5 Run the Python probe to discover runtime capabilities

This step creates a capability baseline for the Execute Script environment. It prevents trial and error by confirming what is actually available in the TPS script runtime.

### 5.1 Create a Python probe script in A10 Control

Create a new Execute Script object in A10 Control and upload a probe script, for example `ddos_probe_v2.py`.

The probe sends results to syslog using UDP, so the output becomes visible in `/var/log/tps.log` on the Ubuntu server.

The probe script can be found here: [ddos_probe_v2.py](scripts/ddos_probe_v2.py)


### 5.3 Attach the probe to a Violation Action

At A10 Control, create a Violation Action with Execute Script  and point it to the probe script. Attach it to a test threshold that you can trigger repeatedly.

### 5.4 Validate probe output on Ubuntu

On the Ubuntu syslog server, confirm the output appears in the TPS log file.

sudo tail -f /var/log/tps.log | grep DDOS_PROBE_V2

### 5.5 Use probe results as a hard constraint

Use the probe output to decide what is allowed in your examples.

1. Use Python standard library only when third party modules are missing.
2. Avoid shell parsing if grep and sed are not available.
3. Base all example triggers on observed DDOS event values.

sudo tail -f /var/log/tps.log

# Troubleshooting
**1. No syslog received**
1. Validate TPS config logging host <ubuntu-ip> use-mgmt-port
2. Use tcpdump on Ubuntu
3. Validate firewall rules
```bash
sudo tcpdump -ni any port 514
sudo ufw status
```
**2. Syslog packets arrive but /var/log/tps.log stays empty**
1. Validate rsyslog config
2. Check rsyslog service logs
3. Confirm the TPS source IP matches your filter in /etc/rsyslog.d/30-tps.conf
```bash
sudo rsyslogd -N1
sudo journalctl -u rsyslog -n 100 --no-pager
```
**3. Watcher runs but produces errors**
1. Check service logs
2. Validate AXAPI reachability
3. Validate AXAPI auth endpoint
```bash
sudo journalctl -u ddos-watch -n 200 --no-pager
curl -k -i https://<<mitigator-mgmt-ip>>/axapi/v3/version
curl -k -i -H 'Content-Type: application/json' -d '{"credentials":{"username":"admin","password":"<PASSWORD>"}}' https://<<mitigator-mgmt-ip>>/axapi/v3/auth
```
**4. Too many snapshots**
1. Increase MIN_SECONDS_BETWEEN_SNAPSHOTS
2. Reduce TRIGGERS to only what you need, for example only move to level 3 and Status: Success

**Output files**
1. `/var/log/tps.log`
Raw syslog lines received from TPS

2. `/var/log/ddos_watch_human.log`
Human readable timeline where every entry begins with a timestamp

3. `/var/log/ddos_script_observability.jsonl`
Structured snapshot records (JSONL) including parsed event fields and AXAPI output snippets
