## How to use `bgp_inject_withdraw.py` in the zone Violation Action configuration

The combined script decides what to do based on `DDOS_EVENT`.

1. If `DDOS_EVENT` equals `DDOS_ZONE_ESCALATION`, it injects the BGP prefix
2. If `DDOS_EVENT` equals `DDOS_ZONE_DE_ESCALATION`, it withdraws the BGP prefix
3. For any other value it logs and exits without changes

Because we measured that lifecycle events are delivered on different zone level hooks, the safest configuration is to attach the same combined script to multiple zone level attachment points.

### Recommended attachment strategy

Our measurements showed the following pattern in the ICMP zone test.

1. `DDOS_ZONE_ESCALATION` appeared on L1 and L2 during scale up
2. `DDOS_ZONE_DE_ESCALATION` appeared on L3 when stepping down from level 3 to level 2
3. `DDOS_ZONE_DE_ESCALATION` also appeared on L2 and L1 during further step down

Therefore, for a clean full lifecycle behavior, attach the combined script as follows.

1. Attach the combined script at L1 zone level
2. Attach the combined script at L2 zone level
3. Attach the combined script at L3 zone level

The script is idempotent, so running it multiple times is safe. If the prefix is already present, inject will do nothing. If the prefix is already removed, withdraw will do nothing.

### Step by step configuration

1. Upload `bgp_inject_withdraw.py` as an Execute Script object in A10 Control
2. Create a Violation Action profile that uses Execute Script and selects `bgp_inject_withdraw.py`
3. In the ICMP zone configuration, assign that Violation Action profile to the zone level hooks for L1, L2, and L3
4. Disable any previous inject or withdraw Violation Actions during the test to avoid conflicts

### How to test and verify

1. Trigger the ICMP test and scale up to level 3
2. Stop traffic and allow the zone to step down to level 0

Verify in two places.

First, syslog execution trace on Ubuntu.

Run:

```bash
sudo tail -f /var/log/tps.log | grep DDOS_BGP_BIDIR
```

Expected output pattern per run.

1. stage=start includes the incoming DDOS_EVENT and the decided action

2. stage=action appears only when the script injects or withdraws

3. stage=end confirms ok true or false and includes verification flags

Second, BGP state verification.

Use either:

1. show ip bgp and show running-config | section router bgp

2. Or your ddos_watch human log, which already reports in_bgp and in_cfg

**How to interpret results**

1. If you see DDOS_ZONE_ESCALATION and action inject, the prefix should appear in BGP

2. If you see DDOS_ZONE_DE_ESCALATION and action withdraw, the prefix should disappear from BGP

3. If you see only start and end with note skipped, the script was executed but the event did not match inject or withdraw, which is expected in some transitions

**After validation**

If you want fewer script executions, you can reduce the number of zone level attachments.

Only do this after you have confirmed, with probes, which level hooks deliver the lifecycle events in your environment.
