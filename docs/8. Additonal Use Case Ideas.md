# Additional Execute Scripts Use Cases

Below use cases are not worked out validated and tested use cases, but definitly worth a read through.

Though the BGP Network Violation Action use case is the biggest, execute scripts can also be used for several other useful purposes, as long as you stay within your known constraints: Python standard library only, limited shell tooling, output through syslog, and event delivery depending on the attachment point.

Here are some practical categories with concrete examples.

# 1. Observability and Operations

This is almost always useful and low risk.

## Incident journaling

On escalation and de-escalation, write a clean syslog entry such as `incident start` and `incident end`, including `run_id`, `zone`, `protocol`, and timestamps. This is useful for NOC handover and post-incident review.

## Triggering state snapshots

You can use the script run as a marker so that external tooling knows exactly when to pull snapshots through AXAPI, similar to `ddos_watch`.

# 2. Policy toggles on the TPS itself

Think of this as setting a knob rather than changing routes.

## Enable or disable specific mitigation features per phase

For example, on escalation you could enable an additional policy or activate a stricter rate limit. On de-escalation, you revert it again. This must be done very carefully and idempotently, and always verified.

# 3. Blacklist or reputation actions

If the TPS can expose top talkers through an external API call, you can build a workflow around that.

## Example

On escalation, you trigger an external AXAPI call to retrieve top sources and then apply a temporary blacklist or ACL update. In practice, this often fits better in an external controller than inside the script runtime itself.

# 4. Service protection mode switch

For example, activating a different profile or template.

## Example

In some environments, you may want to switch to a more aggressive profile during a major attack. If ACOS supports this through configuration switching, it can be automated. This is more complex and requires strong guardrails.

# 5. Integration use cases

Not necessarily from within the TPS script runtime itself, but from your syslog controller pattern.

## Examples

- Cloud scrubbing signal.
- Ticketing or ChatOps.
- Webhook to SOC tooling.

These are often very practical and deliver value without having to touch routing.

# 6. Forensic or compliance logging

This is often important in banking and other regulated environments.

## Example

On escalation, log a minimum set of metrics that you may need later: zone, start time, event type, and thresholds. This can create an audit trail even without a SIEM.

# Is BGP the most important one

For traffic steering toward scrubbing, BGP is often the primary use case. It is the most direct way to redirect traffic, and it works independently of the provider as long as you can speak BGP.

However, the second most important category is usually observability and automation glue. In other words, scripts that mainly signal, mark events, and trigger external tooling. That is often much easier to manage than deep configuration changes.
