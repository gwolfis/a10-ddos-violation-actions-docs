## Use case: Cloud scrubbing signal (API or webhook integration)

This use case describes a “cloud scrubbing signal” workflow. The goal is to notify an external cloud scrubbing provider or orchestration platform when an attack starts or ends.

A cloud scrubbing signal is **not** the same as BGP diversion.

### How this differs from the BGP use cases

**BGP inject/withdraw**  
- Changes routing directly by advertising or withdrawing a prefix  
- The traffic path changes immediately based on BGP policy  
- Best when you control BGP advertisement and diversion is achieved via routing

**Cloud scrubbing signal**  
- Sends an integration signal to an external platform (API or webhook)  
- Does not change routing by itself  
- The cloud provider may start a workflow such as enabling scrubbing, reserving capacity, applying mitigation policies, or activating tunnels

### When a cloud scrubbing signal is useful

1. **Provider workflow is API driven**  
   Some scrubbing services are activated through a portal or API, not through BGP diversion. A signal can start or stop scrubbing, update mitigation policies, or enable GRE/IPsec tunnel handling.

2. **Pre-warming or capacity reservation**  
   You can notify the provider early (for example at a lower severity) so capacity is allocated before you perform a routing change. This can reduce failover time and improve stability.

3. **Customer does not control BGP**  
   In some environments the customer cannot inject routes. The upstream provider performs diversion on their side after receiving a signal. In that model the signal is the only automation path.

4. **Operational context and reporting**  
   The signal can include context such as zone name, protocol, threshold, incident identifier, or timestamps. This helps the provider and SOC/NOC triage faster and supports post-incident reporting.

### Typical trigger conditions

Common triggers for sending a signal are:

- **Start signal** on escalation lifecycle  
  `DDOS_EVENT == DDOS_ZONE_ESCALATION`

- **Stop signal** on de-escalation lifecycle  
  `DDOS_EVENT == DDOS_ZONE_DE_ESCALATION`

In practice, the best attachment points and event delivery should be confirmed with probes, as described earlier in this documentation.

### Common payload fields

A minimal signal payload commonly includes:

- Event type start or stop.  
- Destination zone or service name.  
- Protocol.  
- Timestamp.  
- Optional severity or level.  
- Optional threshold details when available.  

### Note on HTTP webhooks from Execute Script

From our probe results we know the TPS Execute Script runtime includes the Python standard library (for example `urllib` and `ssl`), but it does not include common third party HTTP libraries such as `requests`.

This means:

- It is *technically possible* to implement an HTTP or HTTPS webhook using the Python standard library (`urllib`).
- Whether the webhook will actually work in practice depends on whether **outbound network connectivity is permitted from the TPS Execute Script sandbox**.

In some environments, outbound connections from Execute Script are restricted or blocked. In that case a webhook implementation can be correct but still fail because no outbound packets are allowed.

Recommended pattern when outbound connectivity is uncertain:

- Forward TPS syslog to an external server (Ubuntu).
- Run the webhook sender as an external controller on that server.
- Use syslog events (for example `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`) as triggers for the webhook.

This external controller is typically deployed on the same server that already receives syslog (the `tps.log` host), because it has immediate access to the event stream and avoids limitations of the TPS script runtime.

### Summary

Cloud scrubbing signaling is an integration use case.  
BGP diversion is a routing use case.  
Both can be used together: a signal can prepare the scrubbing platform, while BGP performs the actual diversion when required.
