# Violation Action Scripts: Observability and Runtime Reality (Field Notes)

This page documents practical findings from a lab setup using A10 TPS (Mitigator) + A10 Control, focused on **Violation Action → Execute Script**. The goal is to close gaps in the current documentation by providing a repeatable way to discover what actually works in the script runtime and how to troubleshoot reliably.

## What is missing from the current documentation

The official documentation describes the feature conceptually (Violation Actions, Execute Script, timeouts, example workflows), but it does not provide enough operational detail to make scripting predictable in real deployments.

Key gaps observed in the field:

1. **No clear execution context description**
   The docs do not clearly state what the script runtime environment looks like (available binaries, network capabilities, filesystem access, stdout/stderr handling).

2. **No reliable troubleshooting method**
   In practice, the Mitigator `show log` usually only shows high-level job state:
   `Imported file ...`, `Script ... Running`, `Script ... Success`.
   This does not provide script output, errors, or branch decisions.

3. **Assumptions about Python libraries**
   Example scripts often use libraries like `requests`. In our environment, third-party Python libraries are not present, causing scripts to fail at import time.

4. **Shell scripting expectations**
   The docs do not describe the limitations of the shell environment. Many common tools (grep/sed/awk/etc.) are not available, so typical shell-style parsing and logic does not work.

These gaps lead to trial-and-error deployments. To solve this, we built a probe-driven method to discover runtime capabilities and capture device state at the moment scripts execute.

## How we validated the runtime: probe-based discovery

We used a **Python probe script** executed as a Violation Action “Execute Script”, and forwarded Mitigator logs to an external syslog server on Ubuntu.

The probe sends structured syslog lines reporting:
- Available `DDOS_*` environment variables and their values
- Python interpreter version
- Which Python modules can be imported
- Which shell tools/binaries exist in the execution PATH

This creates an auditable, repeatable capability snapshot that can be used to improve documentation and build compatible scripts.

## Runtime facts discovered by the Python probe

### Python interpreter

The script runtime uses:

- Python version: **3.6.8**
- Invoked via: **/usr/bin/python**

This means:
- Python 2-only modules (`urllib2`, `httplib`) are not available.
- Scripts must be compatible with Python 3.6.

### Environment variables (DDOS_*)

In our ICMP test scenario, the following variables were present:

- `DDOS_ALERT_TYPE`
- `DDOS_DST_NAME`
- `DDOS_DST_PORT`
- `DDOS_EVENT`
- `DDOS_PROTOCOL`
- `DDOS_THRESHOLD`


Using the Python event-capture probes in our lab, we observed that `DDOS_EVENT` is not a single “mode”, but appears as multiple event types. In our ICMP zone test we consistently saw these three values:

- `DDOS_INDICATOR_EXCEED`
- `DDOS_ZONE_ESCALATION`
- `DDOS_ZONE_DE_ESCALATION`

The other `DDOS_*` variables behave differently depending on whether the event is an **indicator event** or a **zone lifecycle event**.

### Observed behavior per variable

**DDOS_EVENT**  
- Observed values (this lab):  
  `DDOS_INDICATOR_EXCEED`, `DDOS_ZONE_ESCALATION`, `DDOS_ZONE_DE_ESCALATION`

**DDOS_THRESHOLD**  
- For `DDOS_INDICATOR_EXCEED`: populated with the indicator threshold value of the level that triggered (e.g. 30 at L1, 50 at L2, 75 at L3).  
- For `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`: reported as `-` (not tied to a specific threshold exceed).

**DDOS_ALERT_TYPE**  
- For `DDOS_INDICATOR_EXCEED`: populated (in our lab: `FAIL`).  
- For `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`: reported as `-`.

**DDOS_DST_NAME**  
- In our lab: consistently `zone1` for all observed event types.

**DDOS_PROTOCOL**  
- In our lab: consistently `icmp` for all observed event types.

**DDOS_DST_PORT**  
- In our lab: consistently `1` for all observed event types (for ICMP this behaves like a placeholder).

### Practical implications for script authors

1. Treat `DDOS_THRESHOLD` and `DDOS_ALERT_TYPE` as optional fields.  
   They are meaningful for indicator events, but may be `-` for zone lifecycle events.

2. Always write scripts defensively.  
   Use defaults for missing values and do not assume every event contains the same context.

3. If your use case depends on thresholds, bind it to `DDOS_INDICATOR_EXCEED`.  
   If your use case depends on escalation lifecycle, bind it to `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`.

### Example summary table from the lab

| DDOS_EVENT | DDOS_ALERT_TYPE | DDOS_THRESHOLD | DDOS_DST_NAME | DDOS_PROTOCOL | DDOS_DST_PORT |
|---|---|---|---|---|---|
| DDOS_INDICATOR_EXCEED | FAIL | 30 / 50 / 75 (depends on level) | zone1 | icmp | 1 |
| DDOS_ZONE_ESCALATION | - | - | zone1 | icmp | 1 |
| DDOS_ZONE_DE_ESCALATION | - | - | zone1 | icmp | 1 |


Important note:
- The number and names of `DDOS_*` variables are **not guaranteed** across builds or configurations. Use a probe to confirm what exists in your environment.

### Python library support

The probe tested imports and found:

**Available (examples):**
- `os`, `sys`, `time`, `datetime`
- `json`, `base64`
- `socket`, `ssl`
- `re`, `subprocess`
- `urllib`
- `ipaddress`

**Not available (examples):**
- `requests`
- `yaml`
- `paramiko`
- `OpenSSL`
- `Crypto`
- `cryptography`
- `netaddr`
- `lxml`

Practical implication:
- Scripts must be written to work with the **Python standard library only**.
- If documentation examples require `requests`, provide an alternate example using `urllib` for HTTP calls.

## Shell / Bash limitations in the Execute Script environment

The probe confirmed that `/bin/sh` and `/bin/bash` exist, but the environment is very minimal.

### Available tools (examples)
- `/bin/sh`
- `/bin/bash`
- `/usr/bin/python`
- `/usr/bin/curl`
- `/usr/bin/ping`

### Missing tools (examples)
Common utilities typically used for scripting and parsing were not present:

- `grep`, `sed`, `awk`, `cut`, `tr`
- `head`, `tail`
- `logger`
- `nc`, `netcat`, `socat`
- `ip`, `ifconfig`, `route`, `ss`, `netstat`
- `wget`
- `busybox`

Practical implication:
- Shell scripts should be treated as **thin wrappers only** (e.g., calling `curl`).
- Avoid shell logic that relies on parsing output, piping, or text transformations.
- Use Python for control flow, parsing, and structured payload handling.

## Recommended documentation additions (proposed)

To make Violation Action scripts usable in real environments, we recommend adding an "Operability" section to the official docs:

1. **Execution environment definition**
   Document what is guaranteed:
   - Python version
   - How environment variables are injected
   - Whether filesystem writes are supported
   - Whether stdout/stderr is captured
   - Network restrictions for outbound connections

2. **Supported runtime dependencies**
   Provide a matrix or guidance:
   - Standard library only by default
   - Third-party libraries are not guaranteed
   - Provide `urllib`-based examples instead of `requests` only

3. **Shell limitations**
   Explicitly warn that common utilities may be missing and that parsing is not feasible in shell.

4. **Troubleshooting reference design**
   Include a recommended approach for observability:
   - Use syslog for event triggers
   - Use AXAPI/AGAPI from an external host to capture state
   - Log results externally (JSONL + human-readable format)

## Next steps

With the runtime constraints known, you can now build compatible Violation Action scripts and validate them via a repeatable test methodology:

- Start with a probe and collect the "capability baseline"
- Build one use case at a time (e.g., BGP inject/withdraw)
- Use external state capture (AXAPI/AGAPI) to verify outcomes
- Document expected behavior per event and mitigation level
