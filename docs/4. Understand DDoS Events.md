## Use case: BGP inject and withdraw driven by `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`

This section documents a key discovery from our probe-based testing:  
`DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION` are valid `DDOS_EVENT` values, but whether you see them depends on **where** you attach the Violation Action script.

The DDoS configuration exposes multiple attachment points, and they behave differently:

- **Threshold attachment points** (for example the ICMP packet-rate zone or source thresholds)  
- **Zone level attachment points** (Violation Actions attached at the zone level per mitigation level)

### Deploy Probe to test/measure events
To be able to tes and measure according to the description below, you need to use a probe and this probe is tailored to track events.

Install the script as a Violation Action to your Zone > Service > Protocol. These tests has this probe installed at level 1, level 2 and level 3.

The event capture script can be found here: [ddos_event_capture.py](scripts/ddos_event_capture.py)

Align the tag naming with the level where the event_probe gets installed.

### What has been measured

Two separate measurements have been performed using Python event-capture probes.

### Measurement A: probes attached to ICMP packet-rate thresholds (zone and source)

**Setup**  
The event probe was attached as a Violation Action script to the ICMP packet-rate thresholds.

**Observed result**  
In this setup, the probe consistently reported only:

- `DDOS_EVENT = DDOS_INDICATOR_EXCEED`

And the context fields were populated:

- `DDOS_THRESHOLD` contained the configured threshold value (per level)
- `DDOS_ALERT_TYPE` contained a value such as `FAIL`

**Interpretation**  
When a script is attached to the packet-rate thresholds, it is executed in an **indicator exceed context**, therefore `DDOS_INDICATOR_EXCEED` is the expected event.

This makes threshold attachment points a good fit for actions that must run exactly when a specific indicator threshold is exceeded.

### Measurement B: probes attached to zone levels (L1, L2, L3)

**Setup**  
The event probes were attached as Violation Action scripts at the **zone level per mitigation level**.

**Observed result**  
In this setup we observed lifecycle events:

- `DDOS_EVENT = DDOS_ZONE_ESCALATION`
- `DDOS_EVENT = DDOS_ZONE_DE_ESCALATION`

**Important detail**  
These lifecycle events were not guaranteed to appear on the same level hook every time. In our ICMP zone test we repeatedly observed:

- During scale up (for example moving to level 2 or level 3)
  - `DDOS_ZONE_ESCALATION` commonly appeared on the L1 and L2 zone level hooks
- During step down from level 3 to level 2
  - `DDOS_ZONE_DE_ESCALATION` appeared on the L3 zone level hook
- During further step down (level 2 to 1, and level 1 to 0)
  - `DDOS_ZONE_DE_ESCALATION` also appeared on L2 and L1 zone level hooks

Also note that for lifecycle events the threshold-related fields were typically not populated:

- `DDOS_THRESHOLD` reported `-`
- `DDOS_ALERT_TYPE` reported `-`

**Interpretation**  
Zone level attachment points provide **zone lifecycle context** rather than indicator exceed context. That is why you see escalation and de-escalation lifecycle events there.

### Practical guidance: how to use these events for BGP use cases

There are two correct ways to build BGP automation, depending on which attachment point you choose.

#### Option 1: use threshold attachment points (indicator based)

- Trigger event: `DDOS_EVENT == DDOS_INDICATOR_EXCEED`
- Best for: “do something when this exact indicator threshold is exceeded”
- Typical use: BGP inject at level 3 when the level 3 indicator threshold triggers

This option is reliable because it is directly tied to a specific threshold exceed.

#### Option 2: use zone level attachment points (lifecycle based)

- Trigger events:
  - `DDOS_EVENT == DDOS_ZONE_ESCALATION` to inject
  - `DDOS_EVENT == DDOS_ZONE_DE_ESCALATION` to withdraw
- Best for: “do something when the zone escalates or de-escalates as part of lifecycle”

This option is the best match for BGP cleanup workflows, because you can explicitly bind withdraw to the de-escalation lifecycle.

### Recommended workflow before enabling BGP actions

Because lifecycle events can show up on different zone level hooks depending on the transition, the safe workflow is:

1. Attach event-capture probes to zone levels L1, L2, L3 (unique tags)  
2. Run one full test flow  
   - scale up to level 3  
   - stop traffic and wait until the zone returns to level 0  
3. Observe which zone level hooks receive `DDOS_ZONE_ESCALATION` and which receive `DDOS_ZONE_DE_ESCALATION`  
4. Attach `bgp_inject.py` to the zone level hook that consistently receives `DDOS_ZONE_ESCALATION`  
5. Attach `bgp_withdraw.py` to the zone level hook that receives `DDOS_ZONE_DE_ESCALATION` for the step-down transitions you want to clean up  
6. If you need cleanup at multiple transitions, attach withdraw to multiple zone levels or use the combined script at multiple levels

### Summary

- Threshold attachment points (zone or source packet-rate thresholds) produce `DDOS_INDICATOR_EXCEED` with populated threshold and alert context  
- Zone level attachment points produce lifecycle events `DDOS_ZONE_ESCALATION` and `DDOS_ZONE_DE_ESCALATION`, often with threshold and alert fields as `-`  
- Use probes to determine which events appear at which attachment points in your environment  
- Choose threshold-based triggers for threshold-based actions, and zone-level triggers for lifecycle-based actions such as BGP cleanup
